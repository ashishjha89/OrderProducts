# GraphQL Federation Learning Plan

This document is a hands-on learning guide for GraphQL Federation concepts, applied to the OrderProducts practice project. It is structured for an Engineering Manager leading a Services team at eBay, whose team builds GraphQL subgraphs that plug into a federation gateway owned by a separate Platform team.

---

## The Org Boundary (Why This Matters)

eBay has two distinct teams involved in GraphQL:

```
Client (mobile / web)
        │
        ▼
  Apollo Router / Gateway          ← Platform & Frameworks team owns this
  (supergraph, query planning,
   auth enforcement, rate limiting,
   caching, observability)
        │
   ┌────┴──────────────┐
   ▼                   ▼
product-service    order-service   ← Your Services team owns these
(subgraph)         (subgraph)
```

**Platform team owns**: the router, supergraph composition tooling, query registration, global security enforcement, and observability at the gateway layer.

**Services team (you) owns**: subgraph schema design, entity definitions, resolver implementation, subgraph-level performance, field-level auth context, and schema evolution without breaking client contracts.

Everything in this learning plan is about what happens on the right side of that boundary — what your team builds and is responsible for.

## Core Concepts

Before the step-by-step practice plan, these are the foundational concepts you will encounter at every step.

### What is GraphQL Federation?

Standard GraphQL gives you one schema served by one service. As your API grows, you want different teams to own different parts of the schema independently — without coordinating on a single monolithic schema file.

Federation solves this. Each team owns a **subgraph** — a GraphQL service that defines the types and fields it is responsible for. A **router** (Apollo Router, Cosmo Router) sits in front of all subgraphs, composes their schemas into a single unified **supergraph**, and routes incoming queries to the right subgraphs transparently.

A client hits one endpoint (`/graphql` on the router) and asks for, say, an order with full product details. The router internally fans out: it calls `order-service` for the order, then calls `product-service` to fill in product details, then stitches the result back into a single response. The client never knows this happened.

```
Client query:
  { orders { id lineItems { skuCode product { name price } } } }

Router plan:
  1. Call order-service → get orders + skuCodes
  2. Call product-service._entities(skuCodes) → get Product details
  3. Merge and return unified response
```

### Subgraph vs Supergraph

| Term | Meaning |
|---|---|
| **Subgraph** | An individual GraphQL service (like `product-service`). Owns a slice of the total schema. |
| **Supergraph** | The composed, unified schema that clients see. Built by the router from all subgraph schemas. |
| **Supergraph SDL** | A compiled SDL file (generated by `rover supergraph compose`) that the router loads at startup. |

### The `@key` Directive and Entities

An **entity** is a type that can be referenced and extended across subgraphs. You declare a type as an entity by giving it a `@key` — a field (or set of fields) that uniquely identifies an instance of that type.

```graphql
# In product-service schema
type Product @key(fields: "id") {
  id: String!
  name: String!
  description: String!
  price: BigDecimal!
}
```

`@key(fields: "id")` means: "A `Product` can be uniquely identified by its `id`. If another subgraph has a `Product` with just an `id`, the router knows it can call `product-service` to fill in the rest of the fields."

A type can have multiple keys (e.g., `@key(fields: "id") @key(fields: "skuCode")`), which gives the router more options when planning how to resolve a query.

### The `_entities` Resolver

This is the mechanism behind federation entity resolution. When the router needs to fill in fields from a subgraph (e.g., get `name` and `price` for a list of Products identified by their `id`), it calls a special query on that subgraph:

```graphql
query {
  _entities(representations: [
    { __typename: "Product", id: "prod-1" },
    { __typename: "Product", id: "prod-2" }
  ]) {
    ... on Product { name price }
  }
}
```

You (the subgraph author) implement this resolver. It receives a list of **representations** (an object with `__typename` + key fields) and returns the fully resolved entities.

This is an internal query — clients never call `_entities` directly. The router calls it automatically as part of query planning. But your team is responsible for implementing it correctly.

### The `_service { sdl }` Query

Every federation-compliant subgraph must also implement this query:

```graphql
type Query {
  _service: _Service!
}

type _Service {
  sdl: String!
}
```

The router calls `_service { sdl }` on each subgraph at startup (or on schema refresh) to fetch its SDL. This is how the router discovers what types and fields each subgraph owns, so it can compose the supergraph.

> Note: `order-service` already has `_service: String` as a dummy query stub in its schema. That stub exists because GraphQL requires at least one field in `Query`, but it is not the real federation `_service` contract. Step 1 replaces it.

### `@external` — Referencing a Type You Don't Own

When `order-service` needs to link an order line item to a `Product`, it does not own the `Product` type — `product-service` does. `order-service` can still reference `Product` by declaring a stub with `@external`:

```graphql
# In order-service schema
type Product @key(fields: "id") @external {
  id: String!
}

type OrderLineItem {
  skuCode: String!
  quantity: Int!
  price: BigDecimal!
  product: Product   # router will call product-service to fill this in
}
```

`@external` tells the router: "I know this type exists and I know its key, but I don't own its fields. When resolving `OrderLineItem.product`, go to product-service."

### Apollo Router and the `rover` CLI

**Apollo Router** is the Rust-based gateway that implements the GraphQL Federation v2 spec. It:
- Loads the supergraph SDL at startup
- Receives all client queries at a single endpoint
- Plans how to split each query across subgraphs
- Calls subgraphs, merges responses, and returns a single result to the client

**`rover`** is the CLI tool that composes subgraph schemas into the supergraph SDL file that the router loads. You point it at each subgraph (with its schema URL or SDL file) and it produces `supergraph.graphql`:

```bash
rover supergraph compose --config supergraph.yaml > supergraph.graphql
```

`rover` also performs **schema checks** — it validates that a proposed schema change to one subgraph does not break the supergraph or clients. The Platform team integrates this check into CI so your team cannot deploy a breaking schema change without it being caught first.

### Union Result Types for Errors

The standard GraphQL error model (throwing exceptions that appear in the `errors` array of the response) works, but has a weakness: the error structure is generic and untyped. Mobile clients must parse the `extensions.code` string to know what happened.

A better pattern for mutations is returning a **union result type**:

```graphql
type Mutation {
  placeOrder(input: PlaceOrderInput!): PlaceOrderResult!
}

union PlaceOrderResult = PlacedOrder | InventoryNotAvailableError | ValidationError

type InventoryNotAvailableError {
  skuCode: String!
  message: String!
}

type ValidationError {
  message: String!
  field: String
}
```

Clients can now pattern-match on the concrete type:

```graphql
mutation {
  placeOrder(input: ...) {
    ... on PlacedOrder { orderId orderNumber }
    ... on InventoryNotAvailableError { skuCode message }
    ... on ValidationError { message field }
  }
}
```

This is typed at the schema level — code generators for mobile clients (Swift, Kotlin) produce exhaustive pattern matching. No more `if error.code === 'INVENTORY_NOT_IN_STOCK'` string comparisons.

### The N+1 Problem

This is the most common production performance bug in GraphQL resolvers. It emerges from how resolvers compose.

Consider: a `products` query returns a list of products. Each `Product` has a `category` field that requires a database lookup. A naive resolver looks up one category per product:

```
GET /graphql products { name category { name } }
→ SELECT * FROM products          (1 query)
→ SELECT * FROM categories WHERE id = 1   (1 query per product)
→ SELECT * FROM categories WHERE id = 2
→ SELECT * FROM categories WHERE id = 3
...
→ 1 + N queries total
```

With 100 products, this is 101 database queries for one GraphQL request. This is the N+1 problem.

The fix is **DataLoader / BatchLoader**: instead of resolving each category immediately, defer all category lookups until the product list is complete, then batch them into one query:

```
→ SELECT * FROM products           (1 query)
→ SELECT * FROM categories WHERE id IN (1, 2, 3, ...)   (1 batched query)
= 2 queries total
```

Spring for GraphQL has built-in support for this via `@BatchMapping`. The `@BatchMapping` annotation replaces `@SchemaMapping` and automatically batches all calls within a single request into one method invocation.

### Schema Deprecation

GraphQL schemas are public contracts. Removing or renaming a field breaks every client that depends on it. The safe process is:

1. Add the new field alongside the old one
2. Mark the old field `@deprecated` with a reason
3. Wait for all clients to migrate
4. Remove the old field (after checking usage in the router's analytics)

```graphql
type Product {
  name: String! @deprecated(reason: "Use displayName. Will be removed after 2027-01.")
  displayName: String!
}
```

The `rover schema check` command (run in CI) will flag if a PR tries to remove a non-deprecated field or a field that still has active client usage.

## Learning Path: 6 Steps in This Project

Each step builds on the previous one. Do them in order.

---

### Step 1: Make Both Services Federation-Compliant Subgraphs

**Concept practiced**: `@key`, `_entities` resolver, `_service { sdl }` query, federation spec

**Why first**: Everything else in federation depends on this. Until your services implement the federation subgraph spec, they are just two independent GraphQL APIs with no ability to reference each other.

**What to do in `product-service`**:

1. Add the `com.apollographql.federation:federation-graphql-java-support` library (or switch to the DGS Framework which has built-in federation support).

2. Update `schema.graphqls` to declare `Product` as a federation entity:
   ```graphql
   type Product @key(fields: "id") {
     id: String!
     name: String!
     description: String!
     price: BigDecimal!
   }
   ```

3. Implement the `_entities` resolver — a method that receives a list of `{__typename, id}` representations and returns fully resolved `Product` objects. Spring for GraphQL / DGS generates most of this boilerplate.

4. The `_service { sdl }` query is also auto-generated by the federation library.

**What to do in `order-service`**:

1. Add the same federation library.

2. Replace the dummy `_service: String` query stub — the federation library auto-generates the correct `_service { sdl }` query.

3. Declare `PlacedOrder` as a federation entity keyed by `orderId`:
   ```graphql
   type PlacedOrder @key(fields: "orderId") {
     orderId: String!
     orderNumber: String!
   }
   ```

**How to verify**: After adding the library, call `GET /graphql` with query `{ _service { sdl } }` on each service. The response should return the full SDL of that subgraph. This is the router's discovery mechanism — if this works, the router can compose your service into the supergraph.

### Step 2: Add a Cross-Service Entity Reference

**Concept practiced**: `@external`, cross-subgraph field resolution, the router's query planning

**Why second**: This is the core value of federation — the moment where a single client query transparently reaches across two services. Until you do this, federation is just extra boilerplate with no payoff.

**What to do**:

The domain link already exists: `OrderLineItemInput.skuCode` references a product. After Step 1, you can surface the full `Product` on an order line item.

1. Add a `PlacedOrderDetail` type in `order-service` that includes line items:
   ```graphql
   type Query {
     order(orderId: String!): PlacedOrderDetail
   }

   type PlacedOrderDetail @key(fields: "orderId") {
     orderId: String!
     orderNumber: String!
     lineItems: [OrderLineItem!]!
   }

   type OrderLineItem {
     skuCode: String!
     quantity: Int!
     price: BigDecimal!
     product: Product
   }

   # Stub — order-service only knows the id, product-service fills the rest
   type Product @key(fields: "id") @external {
     id: String!
   }
   ```

2. Implement the `order` query resolver and `OrderLineItem.product` resolver in Java. The product resolver just needs to return an object with the product's `id` — the router will call `product-service._entities` to fill in `name`, `price`, etc.

**How to verify**: Without the router yet, call each service's `_service { sdl }` and inspect the schemas. The relationship is defined but not yet queryable end-to-end. That happens in Step 3.

**Key insight to internalize**: `order-service` never calls `product-service` directly. It just returns a stub with `id`. The router is responsible for the cross-service call. This is why federation scales: services remain decoupled.

### Step 3: Add Apollo Router to Docker Compose

**Concept practiced**: Supergraph composition, `rover` CLI, Apollo Router, the full federation request lifecycle

**Why third**: This is where the concepts from Steps 1 and 2 become observable. For the first time, you can send a single query to one endpoint and watch it resolve across two services.

**What to do**:

1. Install `rover` CLI locally:
   ```bash
   curl -sSL https://rover.apollo.dev/nix/latest | sh
   ```

2. Create `infrastructure/supergraph.yaml`:
   ```yaml
   federation_version: =2.6.0
   subgraphs:
     product-service:
       routing_url: http://product-service:8080/graphql
       schema:
         subgraph_url: http://product-service:8080/graphql
     order-service:
       routing_url: http://order-service:8080/graphql
       schema:
         subgraph_url: http://order-service:8080/graphql
   ```

3. Compose the supergraph SDL:
   ```bash
   rover supergraph compose --config infrastructure/supergraph.yaml > infrastructure/supergraph.graphql
   ```
   Open `supergraph.graphql` and read it. You will see how the router has merged both schemas, resolved the `Product` entity ownership, and annotated each field with which subgraph provides it. This file is the router's "flight plan."

4. Add Apollo Router to `infrastructure/docker-compose.yml`:
   ```yaml
   apollo-router:
     image: ghcr.io/apollographql/router:v1.52.0
     ports:
       - "4000:4000"
     volumes:
       - ./supergraph.graphql:/dist/config/supergraph.graphql
       - ./router.yaml:/dist/config/router.yaml
     command: --supergraph /dist/config/supergraph.graphql --config /dist/config/router.yaml
   ```

5. Create a minimal `infrastructure/router.yaml`:
   ```yaml
   sandbox:
     enabled: true
   ```
   This enables the Apollo Sandbox UI at `http://localhost:4000` for interactive query testing.

**How to verify**: Send this query to `http://localhost:4000/graphql`:
```graphql
{
  order(orderId: "some-id") {
    orderId
    lineItems {
      skuCode
      product { name price }
    }
  }
}
```
The router resolves it: `order-service` returns the order and skuCodes, then the router calls `product-service._entities` for the Product details, then merges and returns the unified response. Watch the logs of both services to see both receiving requests from the same client query.

### Step 4: Model Errors as Union Result Types

**Concept practiced**: Union types, typed error modeling, inline fragments, better client contracts

**Why fourth**: Once you have a working federation, the next most impactful thing is getting the error model right. Your Services team ships a client-facing API — how errors are represented directly affects mobile and web client development experience.

**What to do in `order-service`**:

1. Update `schema.graphqls` to replace the current exception-based error flow with a union result:
   ```graphql
   type Mutation {
     placeOrder(input: PlaceOrderInput!): PlaceOrderResult!
   }

   union PlaceOrderResult = PlacedOrder | InventoryNotAvailableError | ValidationError

   type PlacedOrder @key(fields: "orderId") {
     orderId: String!
     orderNumber: String!
   }

   type InventoryNotAvailableError {
     skuCode: String!
     message: String!
   }

   type ValidationError {
     message: String!
   }
   ```

2. Update `OrderGraphQLController.placeOrder` to return a sealed type / discriminated union in Java instead of throwing exceptions. The resolver returns the appropriate concrete type based on what `OrderService` reports.

3. Remove `GraphQLExceptionHandler` (or reduce its scope to only unexpected system errors, not business errors like inventory failure).

**How to verify**: Send a `placeOrder` mutation with a skuCode that has no inventory. The response should be:
```json
{
  "data": {
    "placeOrder": {
      "__typename": "InventoryNotAvailableError",
      "skuCode": "SKU-001",
      "message": "SKU-001 is not in stock"
    }
  }
}
```
Note: `errors` array is empty. The business error is in `data`, typed and structured. The client knows exactly what happened without parsing error extension strings.

**What to do in `product-service`**:

Apply the same pattern to `createProduct`:
```graphql
type Mutation {
  createProduct(input: CreateProductInput!): CreateProductResult!
}

union CreateProductResult = CreatedProduct | ValidationError
```

### Step 5: Solve the N+1 Problem with `@BatchMapping`

**Concept practiced**: N+1 problem, DataLoader pattern, `@BatchMapping` in Spring for GraphQL, batched entity resolution

**Why fifth**: N+1 is invisible during development and catastrophic in production. You need to experience causing it and then fixing it, so you can spot it in code review permanently.

**What to do**:

1. Add a `reviews` field to `Product` (or a `category` sub-object — any field that requires an additional data fetch per product). Implement it naively with `@SchemaMapping`:
   ```kotlin
   @SchemaMapping(typeName = "Product", field = "category")
   suspend fun category(product: ProductResponse): CategoryResponse {
       return categoryService.getById(product.categoryId) // one DB call per product
   }
   ```

2. Call `{ products { name category { name } } }` and add logging to `getById`. Observe that with 10 products, `getById` is called 10 times.

3. Replace the resolver with `@BatchMapping`:
   ```kotlin
   @BatchMapping(typeName = "Product", field = "category")
   suspend fun category(products: List<ProductResponse>): Map<ProductResponse, CategoryResponse> {
       val ids = products.map { it.categoryId }
       val categories = categoryService.getByIds(ids) // one batched call
       return products.associateWith { product ->
           categories.first { it.id == product.categoryId }
       }
   }
   ```

4. Repeat the same query. `getByIds` is now called once, regardless of list size.

**How to verify**: Add a `Thread.sleep(50)` (or Kotlin `delay(50)`) inside both the single and batch fetchers to simulate DB latency. A list of 20 products takes ~1000ms with the naive approach (20 × 50ms sequential) and ~50ms with `@BatchMapping` (one call). The difference is measurable with GraphiQL response timing.

**Connection to federation**: The `_entities` resolver you wrote in Step 1 is itself a batched endpoint — the router always sends a list of representations in a single call. This is why federation's cross-service resolution is efficient: it is batch-by-design at the protocol level.

### Step 6: Schema Evolution and Deprecation

**Concept practiced**: `@deprecated` directive, breaking vs non-breaking changes, schema governance, `rover schema check`

**Why last**: Deprecation only matters once you have a real schema worth evolving. After building Steps 1–5, your schema has substance — this step teaches you how to change it safely.

**What to do**:

1. In `product-service`, add a `displayName` field to replace `name`:
   ```graphql
   type Product @key(fields: "id") {
     id: String!
     name: String! @deprecated(reason: "Use displayName instead. Will be removed after 2027-Q1.")
     displayName: String!
     description: String!
     price: BigDecimal!
   }
   ```

2. In the resolver, make `displayName` return the same value as `name` (or a reformatted version). Both fields work during the migration window.

3. Run `rover schema check` against the supergraph to see what it reports about the deprecation. This is the CI step the Platform team runs on every subgraph PR.

4. Next, try an actually **breaking** change — remove `description` without deprecating it first:
   ```graphql
   # Remove description from Product
   type Product @key(fields: "id") {
     id: String!
     name: String!
     price: BigDecimal!
   }
   ```
   Run `rover schema check` again. It will refuse the compose if any registered client query references `description`. This is the schema governance guardrail the Platform team enforces.

**What to internalize**: The Platform team's `rover schema check` CI step is your team's safety net, but it is not magic. It only catches clients that have registered their queries (Step 6 of the recommended eBay workflow, query registration). Unregistered queries in REST-style free-form usage are invisible to the check. This is why the Platform team promotes **persisted/registered queries** — and why your Services team needs to understand that clients should pre-register queries, not send arbitrary query strings.
